# Research: Drawing Canvas

## Decisions

### 1. Canvas Lifecycle: Per-Round

- **Decision**: Canvas is created at round start, cleared (discarded) at round transition.
- **Rationale**: Each round has a new secret word, so the drawer needs a fresh canvas. Automatic clear avoids confusion and ensures all guessers see the same state.
- **Alternatives considered**: Per-game canvas (strokes accumulate across rounds — confusing for new rounds), manual-only clear (drawer could forget).

### 2. Stroke Identity: Client-Generated UUID

- **Decision**: Each stroke gets a UUID generated by the drawer's browser before submission.
- **Rationale**: Enables idempotent stroke addition — if a POST is retried, the server can deduplicate by UUID. Guessers can track which strokes are new without server coordination.
- **Alternatives considered**: Sequential index (fragile on reorder), server-assigned ID (adds round-trip latency for ID generation).

### 3. Stroke Completion Trigger: Mouse-Up

- **Decision**: A stroke is considered "complete" when the user lifts the mouse button.
- **Rationale**: Simplest model for HTTP polling — only complete strokes are sent. No need for partial stroke streaming (which would require WebSockets or very short polling intervals).
- **Alternatives considered**: Time-based periodic flush (more network calls, partial strokes visible), both mouse-up + periodic (over-engineered for polling).

### 4. Poll Failure UX: Subtle Indicator

- **Decision**: Show a small non-blocking "connection issue" indicator that auto-clears on next successful poll.
- **Rationale**: Drawing sync is non-critical — silent failure would hide problems, persistent error would distract from the game.
- **Alternatives considered**: Silent retry (no feedback), persistent error (distracting).

### 5. Drawing Data Cleanup: End-of-Round

- **Decision**: Previous round's strokes discarded immediately on round transition.
- **Rationale**: Minimizes memory footprint per constitution Principle IV. Strokes from old rounds have no use after transition.
- **Alternatives considered**: End-of-game (more memory, no post-game replay planned).

### 6. Drawing Sync Mechanism: HTTP Polling

- **Decision**: Guessers poll `GET /api/rooms/:code/canvas` every 2 seconds for full CanvasState.
- **Rationale**: Required by constitution Principle III. Full-state fetch is simpler than delta sync for this scale (<20 participants).
- **Alternatives considered**: Delta sync (more complex, unnecessary at this scale).

### 7. Data Format: Vector Strokes

- **Decision**: Store strokes as arrays of point coordinates + metadata, not bitmap images.
- **Rationale**: Efficient incremental updates, small payload size, resolution-independent.
- **Alternatives considered**: Bitmap images (large payloads, lossy scaling).

### 8. Canvas Technology: HTML5 Canvas API

- **Decision**: Use native `<canvas>` element with 2D context API for rendering.
- **Rationale**: Fast, well-supported, no external dependencies. React ref for canvas access.
- **Alternatives considered**: SVG (slower for many strokes), third-party library (unnecessary dependency).
